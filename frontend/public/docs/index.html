Architecture · zero-knowledge file sharing

wormhole · share.dwiesta.pro

A wormhole-style file drop where the four-word phrase is the only key — it never leaves the browser, so the servers, the storage, and the operator all hold nothing but ciphertext. Homelab carries the baseline load for free; AWS is rented only when the pipe is full.

client-side AES-256-GCM Argon2id KDF Rust · Axum Redis · TTL 24h private S3 NixOS · nginx homelab → EC2 burst

The property

The four words are the whole secret

Everything follows from one rule: the encryption key is derived from the mnemonic in the browser, and the words are never transmitted. Upload sends ciphertext and a one-way hash; download re-derives the key from the same words locally. No component in the middle can read a file — including the person who runs it.

The plaintext exists in exactly two places: the sender's tab and the recipient's tab. The backend is a ciphertext pipe, Redis holds an anonymous hash → opaque pointer, and S3 stores encrypted blobs. Lose the words and the file is gone — by design, there is no recovery.

Crypto flow

What the browser does, and what it hands over

The distinctive work all happens client-side. The server only ever receives the green-annotated values below — a lookup handle, a public salt, and an encrypted stream.

Threat surface

Who can actually decrypt a file

The honest test of a zero-knowledge claim is a per-component audit. Compromise any single row below — even the operator — and the plaintext still doesn't fall out.

ComponentWhat it seesCan decrypt?
browser (both ends)plaintext + the wordsyes — it's the owner
backendciphertext stream, id, saltno
Redisid → {s3_key, salt, encName}no — no key, name is ciphertext
S3encrypted blob onlyno
operator (you)all of the above at onceno — genuinely can't

The honest weak point: a 4-word phrase is brute-forceable here. Magic-wormhole survives on four words because its live PAKE handshake gives an attacker exactly one guess before the channel dies. This design is store-and-forward — the ciphertext sits in S3 for 24 hours, so guessing can happen offline and at scale.

Mitigation, chosen deliberately: 6 words for entropy, Argon2id to make each guess expensive, and a Redis-backed rate limit on the lookup endpoint so the id can't be hammered. Secret strength is a real security parameter in this system, not a UX detail.

Topology

Homelab is free baseline; AWS is rented burst

Because the backend is stateless — all shared state lives in Redis and S3 — any instance can serve any request. That's the precondition for scaling out. The homelab runs the baseline for free; identical EC2 backends spin up only when the pipe saturates, and stop when it drains.

edge · always on

VPS (NixOS)

Serves the static React app, terminates TLS on *.dwiesta.pro, load-balances the backend pool, and hosts Redis. The one box that sees everything and is never the thing under stress.

router

nginx + mutable include

Nix declares the structure; a single included file lists the live backends. Everything stays declarative except the one line-list that is inherently runtime state.

controller

autoscaler

A small Rust systemd unit — not a microservice. A loop: read concurrency, cross a threshold, call the AWS SDK, rewrite the include, reload nginx. Scales on connections, not CPU.

metadata

Redis

Maps an anonymous hash to an opaque, encrypted pointer with a 24-hour TTL that auto-expires the record. Also holds the lookup rate-limit counters. Never holds a key or a plaintext name.

compute · baseline

homelab backend

The identical Axum binary, run for free. Streams ciphertext to and from S3 using a tightly-scoped IAM user key stored with agenix, alongside your existing secrets.

compute · burst

EC2 backend(s)

The same binary from a baked AMI, started on demand. Uses an IAM instance role — zero secrets to ship, credentials rotate automatically. Joins the WireGuard mesh and the nginx pool, then stops.

Autoscaling

nginx doesn't scale — a controller beside it does

nginx is a reliable, dumb router; it has no idea how to start an EC2 instance or edit its own backend list. So the intelligence is a separate loop that manages nginx. And because the config is generated by Nix and lives read-only in the store, the loop can't hand-edit it — the seam is a single mutable file that Nix includes but never owns.

Cost & data path

Where the money actually goes

  • Compute is the only real spend. Homelab baseline is free; you pay EC2 instance-hours only during bursts, then stop-instances ends the meter.
  • S3 is pennies. Encrypted blobs with a 24-hour lifecycle rule never accumulate; ingress is free, and same-region EC2→S3 egress is free.
  • No presigned URLs. Bytes stream through the backend by choice, so the browser never holds a direct-to-S3 link — the bucket stays fully private, reachable only by IAM-signed calls.
  • Redis stays in the megabytes forever — it holds a few hundred bytes of ciphertext pointer per file, never the file itself.

Delivery

Build order — working demo first

  1. Core loop, homelab only a working app
    React encrypt/decrypt ↔ Axum streams ciphertext ↔ S3 ↔ Redis mapping ↔ word generation & lookup ↔ 24h expiry. End-to-end zero-knowledge, one backend.
  2. Put the edge in front
    nginx include seam + share.dwiesta.pro vhost in Nix, homelab registered in the pool, WireGuard verified, Redis declared in Nix.
  3. Prove balancing
    Backend exposes /metrics; add one manually-started EC2 backend to the pool and confirm requests balance across homelab + cloud.
  4. Automate the burst
    Autoscaler start/stop + upstream rewrite + cooldown. Bake the AMI. Terraform it so the whole AWS side is reproducible.
  5. Chunked streaming encryption phase 2
    Lift the ~100 MB cap: per-chunk nonces and streamed AES-GCM so multi-gigabyte files never exhaust the tab.
